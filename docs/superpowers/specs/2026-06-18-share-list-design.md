# Дизайн: «Поделиться списком»

**Дата:** 2026-06-18
**Статус:** утверждён, готов к планированию реализации

## Цель

Добавить возможность владельцу списка пригласить других пользователей по ссылке. Приглашённые могут совместно работать с содержимым списка (добавлять, отмечать, удалять позиции), но не могут изменять метаданные самого списка или приглашать дальше.

## Сводка ключевых решений

- **Механизм шеринга**: одна многоразовая ссылка-приглашение на список с TTL 24 часа.
- **Уровень доступа участника**: полный доступ к содержимому (items), запрет на rename / delete / share самого списка.
- **Приём приглашения**: получатель обязан зарегистрироваться/войти; гостевой доступ не поддерживается.
- **Хранение ссылки**: одна активная ссылка на список (UPSERT). Создание новой невозможно, пока активна предыдущая.
- **Отзыв ссылки до TTL**: не поддерживается (out of scope).
- **Управление участниками**: владелец видит участников и может удалять; участник может выйти сам. Владелец не может удалить себя через этот же механизм.
- **Дашборд участника**: чужие списки отображаются в той же ленте с пометкой «общий · от {owner.name}».
- **UX точки входа**: меню «три точки» в хедере страницы списка. Для владельца: «Пригласить», «Переименовать», «Удалить». Для участника: «Выйти».
- **Удаление long-press на дашборде убирается**; удаление списка переезжает в это меню.

## Изменения схемы данных

Добавляются две модели в `prisma/schema.prisma`:

```prisma
model ListMembership {
  list      List     @relation(fields: [listId], references: [id], onDelete: Cascade)
  listId    String
  user      User     @relation(fields: [userId], references: [id], onDelete: Cascade)
  userId    String
  joinedAt  DateTime @default(now())

  @@id([listId, userId])
  @@index([userId])
}

model ListInvite {
  list      List     @relation(fields: [listId], references: [id], onDelete: Cascade)
  listId    String   @id
  token     String   @unique
  expiresAt DateTime
  createdAt DateTime @default(now())
}
```

Дополнения существующих моделей:
- `User` — обратная связь `memberships ListMembership[]`.
- `List` — обратные связи `memberships ListMembership[]` и `invite ListInvite?`.

Решения по схеме:
- **Владелец не дублируется в `ListMembership`** — он остаётся отдельной сущностью через `List.ownerId`. Принадлежность «owner / member» определяется одной из двух проверок, что упрощает существующий код владельца.
- **`ListInvite.listId` — первичный ключ** — естественно реализует семантику «одна активная ссылка на список», UPSERT тривиален.
- **Токен — opaque random** (`crypto.randomBytes(32).toString('base64url')`), а не JWT. Удаление строки = инвалидация. Не зависит от секрета.
- **Каскады** на оба отношения: удаление списка или пользователя автоматически чистит membership-ы и инвайты.

Создаётся новая Prisma-миграция; имя выбирает Prisma на момент генерации.

## Авторизация — общие helper-функции

В `lib/access.ts` (новый файл) выносим:

```ts
// owner ИЛИ member
requireListAccess(userId: string, listId: string)
  : Promise<{ list: List, isOwner: boolean } | { error, status }>

// только owner
requireListOwner(userId: string, listId: string)
  : Promise<{ list: List } | { error, status }>
```

Существующие inline-проверки `list.ownerId !== session.userId` в роутах заменяются на вызовы этих функций.

## API

### Изменения существующих эндпоинтов

| Эндпоинт | Сейчас | Станет |
|---|---|---|
| `GET /api/lists` | свои списки | + списки через membership; в каждом элементе появляются поля `owner: {id, name}` и `isOwner: boolean`; сортировка `createdAt desc` через `UNION` или два запроса с merge |
| `GET /api/lists/[id]/items` | owner only | **access** (owner ИЛИ member) |
| `POST /api/lists/[id]/items` | owner only | **access** |
| `PATCH /api/lists/[id]/items/[itemId]` | owner only | **access** |
| `DELETE /api/lists/[id]/items/[itemId]` | owner only | **access** |
| `PATCH /api/lists/[id]` (rename) | owner only | **owner only** (без изменений) |
| `DELETE /api/lists/[id]` | owner only | **owner only** (без изменений) |

`GET /api/lists` response:
```ts
{
  data: Array<{
    id: string
    name: string
    createdAt: string
    owner: { id: string, name: string }
    isOwner: boolean
    _count: { items: number }
  }>
}
```

### Новые эндпоинты

| Метод | Путь | Доступ | Поведение |
|---|---|---|---|
| `GET` | `/api/lists/[id]` | access | Возвращает `{ id, name, isOwner, owner: { id, name } }`. Используется страницей `/lists/[id]` для самостоятельного получения имени и роли (вместо query-параметра `?name=`). |
| `POST` | `/api/lists/[id]/invite` | owner | UPSERT инвайта. Если действующий (не истёк) есть — возвращает его. Иначе генерирует новый токен, `expiresAt = now + 24h`. Возвращает `{ token, expiresAt }`. Полный URL `${origin}/invite/${token}` собирается на клиенте. |
| `GET` | `/api/lists/[id]/members` | access | Возвращает `{ owner: { id, name }, members: [{ id, name, joinedAt }] }`. |
| `DELETE` | `/api/lists/[id]/members/[userId]` | owner ИЛИ `userId === session.userId` | Удаляет `ListMembership`. Используется и для kick-а владельцем, и для self-leave участником. Если owner пытается удалить себя — 400. Удаление несуществующего membership — 404. |
| `POST` | `/api/invite/[token]/accept` | любой залогиненный | Находит инвайт по токену, проверяет TTL. Валидный токен → UPSERT membership (idempotent). Если вызывающий — владелец, membership не создаётся, возвращается 200. Возвращает `{ listId, listName }`. Истёкший — 410 Gone. Несуществующий / список удалён — 404. |

## UI

### Дашборд `/` ([app/page.tsx])

- Long-press для удаления **удаляется**. Карточка реагирует только на короткий tap → переход в список.
- Для карточек с `isOwner === false` под названием появляется строка `«общий · от {owner.name}»` (мелкая, `text-xs text-muted`, с иконкой users слева). Заменяет/дополняет существующий `pluralItems`-блок (формат: «общий · от {name} · N позиций»).
- Сортировка карточек одна на всех (по `createdAt desc`); своих и чужих списков не разделяем.

### Страница списка `/lists/[id]` ([app/(app)/lists/[id]/page.tsx])

- Query-параметр `?name=` **убирается**. Имя и роль читаются через `GET /api/lists/[id]` (React Query).
- В хедере справа от названия появляется кнопка-иконка «три точки» (вертикальные).
- Нажатие открывает **шторку снизу** (стиль текущей `CreateModal`) со списком действий:
  - Для владельца: «Пригласить» → шторка «Поделиться». «Переименовать» → шторка с input. «Удалить» → confirm → DELETE → `router.push('/')`.
  - Для участника: «Выйти» → confirm «Покинуть список «{name}»?» → DELETE membership → `router.push('/')`.

### Шторка «Поделиться»

Открывается по «Пригласить». Видна только владельцу.

- Заголовок: «Поделиться списком».
- Секция «Ссылка»:
  - Подпись `«Действует до DD.MM HH:mm»` (формат `toLocaleString('ru-RU')`).
  - Поле readonly input со значением `${window.location.origin}/invite/${token}`.
  - Кнопка «Скопировать» → `navigator.clipboard.writeText`; показывает «Скопировано» на 2 секунды.
  - При открытии шторки автоматически делается `POST /api/lists/[id]/invite` (UPSERT) и подставляется результат. В состоянии загрузки — скелетон поля.
- Секция «Участники»:
  - Первая строка — владелец с пометкой «владелец», без кнопки удалить.
  - Далее — участники: `{name} · {joinedAt relative}` + крестик справа.
  - Клик по крестику → confirm «Удалить {name} из списка?» → DELETE membership.
- Кнопка «Закрыть» внизу.

Fallback для clipboard: если `navigator.clipboard` недоступен (старые WebView), показываем выделенный текст с инструкцией «скопировать вручную».

### Шторка «Переименовать»

Открывается по «Переименовать». Видна только владельцу.

- Поле input с текущим именем, autofocus.
- Кнопки «Сохранить» (disabled пока пусто/без изменений) и «Отмена».
- На «Сохранить»: `PATCH /api/lists/[id]` → обновляет имя в кэше React Query, закрывает шторку.

## Поток присоединения по ссылке

Маршрут — `/invite/[token]` ([app/invite/[token]/page.tsx], в корне `app/`, без route group). Реализация — серверный компонент (RSC).

### Случай 1: пользователь не залогинен

1. На странице нет middleware-гарда (матчер middleware — `['/', '/lists/:path*']`). Сервер-компонент сам читает сессию через `getSession()`.
2. Если сессии нет → `redirect('/login?next=/invite/{token}')`.
3. Страницы `/login` и `/register` читают query `next`, при успехе делают `router.replace(next ?? '/')`.
4. После входа пользователь возвращается на `/invite/{token}` → переходим к Случаю 2.

### Случай 2: пользователь залогинен

1. Сервер-компонент вызывает внутренний flow accept-а (либо POST в свой же API, либо прямой вызов prisma-логики; решается на этапе реализации, но семантика та же).
2. Успех → `redirect('/lists/{listId}')`.
3. Ошибки рендерятся как полноэкранные сообщения:
   - **Истекла** → «Срок действия ссылки истёк. Попросите владельца сгенерировать новую.» + кнопка «На главную».
   - **Недействительна** (токен не найден или список удалён) → «Ссылка недействительна или список удалён.» + кнопка «На главную».
   - **Уже участник** → не ошибка, тихий редирект на список.
   - **Это твоя ссылка (owner)** → тихий редирект на список без создания membership.

### Изменения в `/login` и `/register`

- Принимают query `next` (string, optional).
- При успехе делают `router.replace(next || '/')` вместо текущего перехода.
- Дополнительные правила: `next` должен начинаться с `/` (открытый redirect недопустим); в противном случае игнорируется.

### Middleware

`middleware.ts` оставляем как есть — invite-маршрут под matcher не попадает и обрабатывает свой flow сам.

### Унификация обработки 403 на клиенте

Существующие fetcher-ы (`fetchLists`, `fetchItems`, `addItem`, `toggleItem`, `deleteItem`, и т.п.) обрабатывают только 401 (редирект на `/login`). После шеринга участник может потерять доступ в процессе работы (kick, leave в другой вкладке, удаление списка владельцем). Все fetcher-ы расширяем: получение 403 → редирект на `/` (дашборд).

## Тестирование

API-тесты (jest + next-test-api-route-handler в `tests/api/`):

- **`POST /api/lists/[id]/invite`**: owner получает токен; member — 403; non-member — 403; повторный вызов возвращает тот же токен; если действующий истёк — возвращается новый.
- **`POST /api/invite/[token]/accept`**: валидный токен создаёт membership; повторный вызов идемпотентен; истёкший — 410; несуществующий — 404; владелец — 200 без membership; неавторизованный — 401.
- **`GET /api/lists/[id]/members`**: owner и member получают данные; non-member — 403.
- **`DELETE /api/lists/[id]/members/[userId]`**: owner удаляет любого члена; member удаляет только себя; owner удаляет себя — 400; member пытается удалить другого — 403; удаление несуществующего membership — 404.
- **`GET /api/lists`**: возвращает свои + те, где есть membership; правильные `isOwner` и `owner.name`.
- **`GET /api/lists/[id]`** (новый): owner и member получают данные с корректным `isOwner`; non-member — 403.
- **`GET /api/lists/[id]/items`** и `POST/PATCH/DELETE` на items: member имеет доступ; non-member — 403.
- **`PATCH /api/lists/[id]`** и **`DELETE /api/lists/[id]`**: member по-прежнему 403.

UI-тестов на компоненты не пишем — это соответствует существующей практике (в `tests/` нет компонентных тестов).

## Edge cases

| Случай | Поведение |
|---|---|
| Владелец удаляет список с участниками | Каскад: `ListMembership`, `ListInvite`, `ListItem` удаляются автоматически. |
| Владелец удалил участника, у которого открыта страница | На следующий API-запрос — 403. Клиентские fetcher-ы (сейчас обрабатывают только 401) расширяем: 403 → редирект на `/`. Без real-time push. |
| Участник вышел сам в другой вкладке | Аналогично — 403 на следующем запросе, редирект на `/`. |
| Удаление аккаунта со связанными items других списков | Существующая проблема FK; **не в скоупе** этой задачи. |
| Часовой пояс TTL | `expiresAt` хранится в UTC, форматируется на клиенте через `toLocaleString('ru-RU')`. |
| Race: два владельца параллельно жмут «Скопировать ссылку» | UPSERT по `listId` (PK) — итог одна строка. Уникальный индекс на `token` исключает коллизии. |

## Безопасность

- Токен — 32 случайных байта в base64url (256 бит). Brute-force невозможен.
- Эндпоинт `accept` требует залогиненной сессии. Анонимный приём не поддерживается.
- Параметр `next` в `/login` и `/register` должен начинаться с `/` (защита от open redirect).
- Токен не логируется (в проекте отсутствует структурный логгер; фиксируем как ограничение на будущее).

## Не входит в скоуп

- Real-time уведомления (новые участники, добавление позиций другими).
- История изменений / аудит-лог.
- Несколько одновременных активных инвайтов на один список.
- Отзыв активной ссылки до истечения TTL.
- Периодическая очистка истёкших инвайтов (удаляются лениво при следующем POST `/invite`).
- Уведомления по email.
- Перенос владения списком другому пользователю.
